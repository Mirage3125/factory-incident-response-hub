# AGENTS.md

## 项目目标

构建可运行、可测试、可用于简历和面试演示的制造业异常处理工作流系统。优先完成真实业务闭环，不以文件数量、页面数量或技术堆叠作为完成标准。

## 必读文件

每次开始任务前，按顺序阅读：

1. `AGENTS.md`
2. `docs/project-spec.md`
3. `docs/implementation-status.md`
4. 与当前阶段相关的现有代码、测试和配置

若文件内容与用户当前指令冲突，以用户当前指令为最高优先级，并在状态文档中记录偏差。

## 阶段边界

- 每次只执行用户明确指定的阶段。
- 不得提前实现后续阶段的大量代码。
- 为当前阶段所必需的最小接口或桩可以创建，但必须真实可运行、有明确用途并在状态文档中说明。
- 不得以“顺便完成”为理由扩大范围。
- 当前阶段验收失败时，先修复，不进入下一阶段。


## 连续执行与硬门禁协议

每个阶段必须同时区分两个结论：

1. **当前阶段交付结果**：`PASS`、`FAIL` 或 `BLOCKED`；
2. **下一阶段放行结果**：`GO` 或 `NO-GO`。

### 核心原则

进入门禁用于发现和修复前置回归，不是让任务提前结束的理由。一次命令失败只是中间状态，不是最终阶段结论。

### 执行规则

1. 开始当前阶段前，运行提示词列出的进入门禁。
2. 如果进入门禁失败：
   - 先定位根因；
   - 修复仓库代码、配置、迁移、容器或脚本；
   - 重新运行失败项以及完整进入门禁；
   - 进入门禁通过后，必须在**同一次任务中继续实现当前阶段**。
3. 不得在修复前置阶段后要求用户“重新发送当前阶段提示词”。
4. `docs/implementation-status.md` 只是记录，不是唯一事实来源：
   - 实际命令全部通过但状态文档过期时，更新文档后继续；
   - 不得仅因文档仍写着 `NO-GO` 就停止。
5. 当前阶段尚未创建相关代码、目录、接口或测试是正常初始状态，正是当前任务要完成的内容，绝不能作为 `BLOCKED` 理由。
6. 测试失败、迁移失败、容器不健康、接口 4xx/5xx、依赖缺失、配置错误、代码缺失或构建失败，默认都属于**可修复问题**。应修复并继续，不得立即结束。
7. 只有出现明确的外部阻塞，才允许最终输出 `BLOCKED + NO-GO`，例如：
   - 必须由用户提供且无法使用 Demo 替代的账号、密钥、许可证或人工审批；
   - Docker Desktop 无法由当前权限启动；
   - 磁盘空间、系统权限或网络故障无法通过仓库修改和已授权命令解决；
   - 继续操作需要破坏用户数据或执行未授权的系统级修改。
8. 在认定外部阻塞前，必须给出根因证据、已尝试的修复、重试结果，以及为什么无法在当前权限内继续。
9. 当前阶段的文件已经创建，不等于阶段通过；退出门禁仍需本次真实命令证据。
10. 只有当前阶段退出门禁全部通过，才能输出 `PASS + GO`。
11. 任何关键验证最终未执行或仍失败时，不得输出 `PASS`。
12. 不允许用“后续阶段再验证”“代码应该可用”“环境受限但实现完成”等措辞绕过验收。
13. 最终报告只能在以下两种情况出现：
    - 当前阶段全部实现并完成退出门禁；
    - 遇到有证据的真实外部阻塞。
   不得把一次中间失败包装成最终报告。

## 修改规则

- 修改前先检查当前目录、Git 状态、现有文档和相关文件。
- 不覆盖或删除与当前任务无关的代码。
- 遵循现有项目结构和命名；确需调整结构时说明原因。
- 文件应职责单一，避免将大量业务逻辑放入路由、n8n Code 节点或单个巨型文件。
- 不创建没有实际用途的空目录、占位类或占位接口。
- 核心功能禁止使用 `TODO`、`pass`、硬编码假结果或无功能按钮冒充完成。
- Demo 模式允许使用确定性模拟数据，但必须能够完成真实调用链和状态变化。

## 实施方法

- 优先采用简单、稳定、适合 Windows 11、PowerShell 和 Docker Desktop 的方案。
- 遵循 YAGNI：没有进入当前验收范围的复杂基础设施不要加入。
- 对关键业务逻辑优先使用测试驱动方式：先写失败测试，再实现最小代码使其通过。
- 所有外部 HTTP 请求必须设置 timeout。
- 所有重试必须有次数上限，并区分可重试与不可重试错误。
- Agent 只提供分析建议；严重等级、状态转换和高风险动作由确定性代码控制。
- RPA 只在 API 因技术故障不可用时降级使用，不能用来掩盖业务参数错误。

## 验证规则

- 每个阶段结束前必须运行与该阶段相关的测试、构建或联调命令。
- 不得在没有实际命令输出的情况下声称“已完成”“已通过”或“可以运行”。
- 如果验证失败，记录失败命令、关键错误、已尝试修复和当前阻塞。
- 不能执行的验证必须明确说明原因，不得伪造结果。
- 优先提供 Windows PowerShell 可直接执行的命令。

## 状态记录

每次任务完成或中断前更新 `docs/implementation-status.md`，至少记录：

- 当前阶段；
- 本次完成内容；
- 创建和修改的文件；
- 实际运行的验证命令；
- 验证结果；
- 已知限制或阻塞；
- 下一阶段允许开始的前置条件。

## 安全规则

- 不得提交 `.env`、API Key、Token、密码、n8n 恢复地址或其他真实凭据。
- 日志必须对密码、Token、模型密钥和恢复地址脱敏。
- `.env.example` 只能包含示例值。
- 不允许 Agent 或 Demo 功能执行真实设备控制。
- 不允许前端获取 n8n `resume_url`。
- 不允许 RPA 日志输出账号密码。
- 完成阶段后检查新文件中是否误写入敏感信息。

## Git 规则

- 不执行 `git push`、发布 Release、创建 PR 或修改远程仓库，除非用户明确要求。
- 不自动提交 Git commit，除非用户明确要求本阶段提交。
- 开始修改前查看 `git status`，避免覆盖用户未提交的更改。
- 禁止使用破坏性命令清除用户修改，例如 `git reset --hard`、`git clean -fd`。

## 端口与环境

默认宿主机端口：

- Frontend：`3100`
- Backend：`8100`
- n8n：`5678`

所有端口必须允许通过 `.env` 覆盖。新增端口前先检查现有配置，避免默认占用 `3000` 和 `8000`。

## 完成报告格式

只有在当前阶段完成退出门禁，或确认存在真实外部阻塞时，才输出最终报告。修复循环中的中间失败不要当作任务结束。

最终回复必须包含：

1. 本次完成内容；
2. 创建或修改的主要文件；
3. 实际执行的验证命令及结果；
4. 未完成内容和已知问题；
5. 进入门禁逐项结果；
6. 退出门禁逐项结果；
7. 当前阶段结论：`PASS`、`FAIL` 或 `BLOCKED`；
8. 下一阶段放行：`GO` 或 `NO-GO`；
9. 只有结论为 `PASS + GO` 时，才可建议用户提交下一阶段；否则只给出解除阻塞步骤。


## 禁止提前结束

以下表达不得作为停止当前任务的理由：

- “前置测试失败，所以请重新发送本阶段提示词”；
- “当前阶段代码尚未创建”；
- “状态文件仍然是 NO-GO”；
- “某个容器第一次启动失败”；
- “某个测试第一次没有通过”；
- “需要先修复上一阶段，因此本次不再继续当前阶段”。

正确行为是：修复 → 重跑进入门禁 → 在同一任务中继续当前阶段 → 执行退出门禁。
